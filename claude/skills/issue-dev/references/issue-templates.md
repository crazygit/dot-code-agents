# Issue 模板

这些模板用于 `issue-dev` 中的 Issue 草稿准备。

原则：

- 先生成草稿，再由用户确认
- 最终创建必须使用 `gh issue create --web`
- 一个 Issue 对应一个明确可交付结果
- 大需求优先拆成父 Issue 加子 Issue，或用 Milestone 聚合

## 单个可交付 Issue

```markdown
## 目标

<一句话说明这次交付要达成什么结果>

## 验收标准

- [ ] <可验证条件 1>
- [ ] <可验证条件 2>

## 技术约束

<会影响实现方式的限制、依赖或排除项；没有可写“无”>

## 交付说明

- 建议分支：`issue-<N>-<slug>`
- 预期 PR 关联：`Fixes #<N>`
- 相关文档：<需要更新的文档，或“无”>
```

## 父级跟踪 Issue

适用于较大需求，需要拆成多个子 Issue 并持续汇总进度。

```markdown
## 目标

<一句话说明整个大需求的最终结果>

## 范围

- 包含：
  - <范围 1>
  - <范围 2>
- 不包含：
  - <非目标 1>

## 子任务

- [ ] #<子 Issue 编号> <子任务标题>
- [ ] #<子 Issue 编号> <子任务标题>

## 验收标准

- [ ] <整体层面的可验证条件 1>
- [ ] <整体层面的可验证条件 2>

## 风险与依赖

- 风险：<风险，或“无”>
- 依赖：<依赖，或“无”>
```

## 子 Issue

适用于父 Issue 已存在时的可独立执行子任务。

```markdown
## 目标

<一句话说明该子任务交付结果>

## 父级关联

- Parent: #<父 Issue 编号>

## 验收标准

- [ ] <子任务可验证条件 1>
- [ ] <子任务可验证条件 2>

## 技术约束

<仅写影响本子任务实现的约束；没有可写“无”>

## 交付说明

- 建议分支：`issue-<N>-<slug>`
- 预期 PR 关联：`Fixes #<N>`
- 需要同步给父 Issue 的内容：<进度、风险、验证结果等>
```

## 标题建议

- 功能：`feat: <交付结果>`
- 修复：`fix: <问题结果>`
- 优化：`enhancement: <优化结果>`
- 杂项：`chore: <维护结果>`

标题要表达结果，不要只写动作。

不推荐：

- `更新接口`
- `处理登录`
- `修一些 bug`

推荐：

- `feat: 支持用户使用邮箱登录`
- `fix: 修复未登录状态下页面跳转死循环`
- `chore: 清理过时的构建配置`

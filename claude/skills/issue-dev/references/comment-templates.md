# 评论与交付模板

这些模板用于 `issue-dev` 的 GitHub Issue / PR 辅助输出。

原则：

- 主 agent 统一决定是否发布
- 创建 Issue 和 PR 仍然必须使用 `--web`，人工最终审核提交
- Issue 评论可以先生成草稿，获得用户授权后再发布

## Kickoff 评论

```markdown
## 启动

- 当前理解：<一句话概括目标>
- 本次范围：<本轮要做什么>
- 暂不处理：<本轮明确不做什么>
- 计划分支：`issue-<N>-<slug>`
- 下一步：<进入设计 / 计划 / 实现>
```

## Plan 评论

```markdown
## 计划

- 实现路径：<简述方案>
- 关键文件：
  - `<path>`
  - `<path>`
- 验证方式：
  - `<test or check>`
  - `<test or check>`
- 风险点：
  - `<risk>`
```

## Progress 评论

```markdown
## 进度

- 已完成：
  - `<done item>`
- 进行中：
  - `<current item>`
- 下一步：
  - `<next item>`
- 当前风险：
  - `<risk or none>`
```

## Blocker 评论

```markdown
## 阻塞

- 阻塞点：<当前被什么卡住>
- 影响：<导致哪些工作无法继续>
- 已尝试：<已经做过哪些尝试>
- 需要输入：<需要用户/团队/外部系统提供什么>
```

## Handoff 评论

```markdown
## 交接

- PR：<链接或草稿状态>
- 代码结果：<本次改了什么>
- 文档结果：<本次更新了哪些文档>
- 测试结果：<测试项和结果>
- 剩余风险：<风险或无>
- 建议下一步：<继续评审 / 验证 / 合并决策>
```

## Completion 评论

```markdown
## 完成

- 验收结果：<已满足项或待确认项>
- 验证方式：
  - `<检查项>`
  - `<检查项>`
- 交付物：
  - `<代码>`
  - `<文档>`
- 是否可关闭：<是/否，以及原因>
```

## PR 描述草稿

```markdown
Fixes #<N>

## 概要

- <变更项>
- <变更项>

## 测试

- <测试项>
- <测试项>

## 备注

- <风险、迁移、发布说明，或无>
```

# 评论与交付模板

这些模板用于 `issue-dev` 的 GitHub Issue / PR 辅助输出。

原则：

- 主 agent 统一决定是否生成和展示发布草稿，最终发布仍由人工确认
- 创建 Issue 和 PR 仍然必须使用 `--web`，人工最终审核提交
- 所有 GitHub 留痕操作都先生成草稿，获得人工审核后再发布
- review 前默认只保留草稿或本地工作日志，不主动发布 kickoff / plan / progress
- kickoff / plan / progress 不贴 commit hash，不写“已完成”结论

## Kickoff 评论

```markdown
## 启动

- 当前理解：<一句话概括目标>
- 本次范围：<本轮要做什么>
- 暂不处理：<本轮明确不做什么>
- 计划分支：`<type>/<N>-<slug>`
- 下一步：<进入设计 / 计划 / 实现>
```

适用条件：

- 默认只生成草稿
- 适合告诉协作者“要开始做什么”
- 不用于同步已写了哪些 commit

## Plan 评论

```markdown
## 计划

- 实现路径：<简述方案>
- 关键文件：`<path>`，`<path>`
- 验证方式：`<test or check>`，`<test or check>`
- 风险点：<risk>
```

适用条件：

- 默认只生成草稿
- 只表达计划，不表达实现完成度

## Progress 评论

```markdown
## 进度

- 阶段性进展：<done item>
- 当前进行中：<current item>
- 下一步：<next item>
- 当前风险：<risk or none>
```

适用条件：

- 仅在用户要求同步、需要跨人协作、或需要公开阻塞前置背景时使用
- 不贴 commit 列表
- 不把本地提交数、临时测试结果直接写成对外承诺

## Blocker 评论

```markdown
## 阻塞

- 阻塞点：<当前被什么卡住>
- 影响：<导致哪些工作无法继续>
- 已尝试：<已经做过哪些尝试>
- 需要输入：<需要用户/团队/外部系统提供什么>
```

## Handoff 评论

```markdown
## 交接

- PR：<链接或草稿状态>
- 代码结果：<本次改了什么，可附已 review 的提交摘要>
- 文档结果：<本次更新了哪些文档>
- 测试结果：<测试项和结果>
- 剩余风险：<风险或无>
- 建议下一步：<继续评审 / 验证 / 合并决策>
```

适用条件：

- PR 已准备好，或已经进入正式 handoff 阶段
- 适合在分支已 push、PR 已创建或即将创建时使用
- 这是第一个明确面向后续评审 / 验证 / 合并动作的外部评论模板

## Review-Ready 评论

```markdown
## 待评审

- 范围：<本轮已完成的范围>
- PR：<链接或草稿状态>
- 已验证：<核心测试或检查>
- 关注点：<reviewer 需要重点看的地方>
- 风险：<risk or none>
```

适用条件：

- 用于“实现已到可评审状态，但 PR 还没最终提交或还没进入 handoff”
- 如果仓库习惯在 PR 前先在 Issue 报备，可优先用这个模板而不是 handoff

## Completion 评论

```markdown
## 完成

- 验收结果：<已满足项或待确认项>
- 验证方式：
  - `<检查项>`
  - `<检查项>`
- 交付物：
  - `<代码>`
  - `<文档>`
- 是否可关闭：<是/否，以及原因>
```

## PR 描述草稿

```markdown
Fixes #<N>

## 概要

- <变更项>
- <变更项>

## 测试

- <测试项>
- <测试项>

## 备注

- <风险、迁移、发布说明，或无>
```
